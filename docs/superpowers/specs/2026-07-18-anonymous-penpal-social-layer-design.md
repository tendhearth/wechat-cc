# 匿名笔友社交层 — 设计 spec(发现层重定向)

> 状态:设计已定稿(经 2026-07-18 一次长讨论)。本文是对 `docs/design/agent-social-network.md`
> 的一次**重大修订 + 重定向**,并给觅食台方向定调。落地计划由 writing-plans 另出(分期,见 §11)。

## 0. 一句话

**产品是一个"陌生/朋友的朋友"的匿名笔友层:我的 CC 和你的 CC 借共同朋友的 CC 牵线,匿名地相遇、写信、交朋友——中间人只转密文、只知道"撮合成了",永远不知道内容,也永远不在系统内交叉真身。想真认识,两人自己在信里换联系方式。**

## 1. 这次重定向改了什么(相对 agent-social-network.md)

原设计笔记把重点放在"熟人层 RPC / 陌生层 Nostr 式网 + web-of-trust 去匿名桥"。讨论后修正为:

- **熟人之间不需要"发现"**——朋友本来就认识,建一条 CC↔CC 的边直接扫码/微信说一声即可,不值得搞协议。熟人边只是底下那张**图**。
- **有价值的是陌生层**:不熟的人借朋友的朋友匿名地发现彼此、通信、交朋友,**拓展人脉**。
- **系统内不做"揭晓真身"**。匿名是永久的 by design;"要认识就自己换联系方式"(微信,或脱微信后的 wechat-cc 好友列表)。→ 最硬的 §8 去匿名桥**被设计掉了**。
- **微信要解耦**:微信只是"建边入口 + 当下的抗女巫信号",不是传输、不是身份。底座 channel-agnostic。

## 2. 铁律 (invariants)

1. **匿名永久**:系统内**只经手假名,真身从不交叉**。要认识 = 两人自己在 E2E 信里换联系方式,系统内容盲、不知情、不参与。
2. **交换意图/情报,不交换原始数据**(沿用原笔记)。CC 说"周二有空",绝不发整张日历。
3. **中间人内容盲**:中间人只路由密文,只拿到**"撮合成功"这个无内容的事实**(温度),不知内容、不知后续。
4. **低打扰**:明信片/信是礼物不是通知;必须 earn presence、知道何时闭嘴。
5. **别做真币**:顶多局部主观信誉,不做可交易 coin、不做全局共识账本。

## 3. 生命周期状态机(身份 × 内容 的可见度)

一条陌生连接的全程,以及"谁看得见什么":

| 阶段 | A↔B 之间 | 中间人 W 看得见 | 传输 |
|---|---|---|---|
| **① 觅食 / 明信片** | 匿名(每段一次性假名) | **粗意图(脱敏)**——它得看见才能判断"我朋友 B 配不配 A" | W 代理 |
| **② 匿名写信** | 匿名假名,两个 CC 交朋友 | **只知"他俩在通信",内容盲(E2E)** | W 只路由密文 |
| **认识(系统外)** | 两人在 E2E 信里自己换微信 / wechat-cc 名片 | **什么都不知道**(连"认识了"都不必知道) | 之后走各自联系方式,系统退场 |

关键细节:
- **匿名是 A 对 B 匿名,不是对 W 匿名**。W 必然认识 A 和 B(定义上他俩都是 W 的朋友)。W 是"两边都认识、但不牵不揭、还内容盲"的可信中介。
- **可见度递减**:撮合阶段 W 看得见粗意图(否则没法判断契合);进入写信后 W 就瞎了(只转密文)。**分界线画在 ①→② 之间**,正是交换 E2E 密钥那一刻。
- **认识不是系统状态**:它是"信的内容",系统内容盲,连"他们认识了"都无需知道。

## 4. 架构分层

### 4.1 传输底座:内容盲信箱 (mailbox)

发现/觅食**不是即时通信**,是异步的(发心愿 → 回音几分钟到几小时慢慢回;写信也异步)。所以底座是**信箱**,不是保活连接:

- **运营模型抄 Tailscale/DERP**:节点**拨出去**连信箱(天然穿 NAT),按 pubkey 认证,信箱**只转密文、看不到明文**,可自建、可多个。能直连的边仍可 `push` 直连。
- **线格式抄 Nostr**:签名事件 + topic tag + (脱微信后)PoW 盖戳;只用它的 **store-and-fetch**(投递/收取/TTL 过期),不挂实时 subscription。
- **拉取模型**:节点**心跳时 poll**(daemon 已有心跳 ticker)。压力极小——无常驻连接、离线天然容忍(信存着下次取)、payload KB 级。粗算 1 万用户每 2 分钟 poll ≈ 83 req/s,单台便宜 VPS 绰绰有余;`brain.youdamaster.cc` 那台当第一个信箱。
- **隐私白送**:信箱**看不到谁在线**(无 presence);poll 加抖动 + 每段一次性假名寻址,时序关联更糊。
- `A2AAgentRecord.transport` 增加 `mailbox` 档;实时那套(语音/伴侣的 YiHub)完全不碰,归另一个子系统。

### 4.2 身份:每段一次性假名

- 每段陌生关系用**一个一次性假名 keypair**(不可关联、最私密)。跨关系不攒全局声誉——声誉走"中间人在自己视角里主观担保"(§10 v0)。
- **wechat-cc 自己的好友列表**是 channel-agnostic 的联系方式基底:"认识"时换的可以是微信,也可以是 wechat-cc 名片。脱微信后仍成立。

### 4.3 熟人边 = 底图(不搞协议)

- 一条 CC↔CC 熟人边 = 双方各自持有对方的假名 pubkey + 信箱地址,存 registry(`transport: mailbox`)。
- 建边靠**带外**:扫码 / 链接 / 微信里说一声。今天微信是最方便的建边入口,但边一旦建立就是 **pubkey↔pubkey,存在 registry,离开微信照样在**。
- 不做熟人"自动发现"协议(讨论确认:熟人不需要发现)。

### 4.4 陌生 / 朋友的朋友 匿名笔友层(产品主体)

- A 的 CC 广播一个匿名意图(心愿),经中间人 W(A 与 B 的共同朋友)转发,触达 2 度外的 B。
- B 的 CC 判断契合 → 回一张匿名明信片("第 2 度的某人 via 老王")。
- 双方都点头 → 经 W 交换**每段一次性假名 pubkey** → 打开 **E2E 笔友通道**。
- 之后 A、B 的 CC 匿名写信;W 只路由密文,只拿"我俩朋友接上头了~"的温度。
- ② 是**终点**:人永不见面也成立。要认识 → 信里自己换联系方式(§3 右列)。

### 4.5 中间人语义

- **内容盲路由器**:撮合阶段看粗意图(脱敏),通信阶段只转密文。
- **温度是它的报酬**:内容无关的一句"你牵成了一对"(§情报经济学的温柔版),这是它愿意帮转的动力。
- **真实好友门 = 抗女巫**:要经 W 撮合,双方得都是 W 的真朋友。这道门在"发现"这一步免费提供女巫抵抗——一千个假号农场没有真朋友,进不来。

## 5. `揭晓` 语义重定向(复用现有机制)

现有 spec #1/#2 的 `揭晓/reveal` 是"双方确认后交叉**真身**"。新愿景下**改指向即可复用**:

> `揭晓` 从"亮真身" → **"双方都愿意开始通信 → 经中间人交换每段一次性假名密钥,打开 E2E 笔友通道"**。

- 双向确认、中间人代理、`social_relay` 两腿 pivot、3-way 温度——**骨架全映射过来**。
- 变的只有:揭晓那一刻交换的是**假名公钥(为了 E2E),不是真名**;且**之后没有下一步去匿名**。
- 需要退场的:现有 reveal 里"persist peer 真实 identity / 真名交叉"的部分——改成交换假名 pubkey + 打开 E2E,真名永不入系统。

## 6. 微信解耦

微信只是两个**可插拔输入**,不是底座:

- **建边入口**之一(今天最方便)——明天可换二维码/链接/别的 IM。边 = pubkey↔pubkey,离开微信仍在。
- **抗女巫信号**之一(今天免费)——真好友关系当女巫抵抗。

底座(信箱 + pubkey 身份 + 转发图 + E2E 信件)**全部 channel-agnostic**。

**decoupling 的代价(要认这笔账)**:微信在时,真好友关系免费扛女巫抵抗;一旦脱微信,新边不再自带真关系担保,得靠 **PoW 地板 + 背书乘数** 补上——即把"抗女巫"从微信手里接管过来。可以晚点付(v1),但要预留位置。

## 7. 防滥用(无稳定声誉下)

每段一次性假名 = 不可关联 = 攒不了全局声誉。所以防滥用是**局部、主观**的三层:

1. **中间人真实好友门**——要滥用你得先是某人真朋友(见 §4.5)。
2. **每个中间人的转发预算/上限**——别让人拿 W 当免费群发管道;W 有可配的转发配额。
3. **两端 CC 各自过滤**——收方 CC 可拒、可静默、可按低打扰铁律节流。

## 8. 与现有代码的关系

| 现有 | 新愿景下 |
|---|---|
| `social_seek` / `social_echo` / 觅食脊柱(spec #1) | 保留,当 ①觅食/明信片 阶段 |
| 转发跳 / `social_relay` / `social_seen_intent`(spec #2) | 保留,当 FoF 撮合 + 中间人代理 |
| `揭晓/reveal` 交叉真身 | **改指向**:交换假名密钥、打开 E2E 通道(§5) |
| 3-way 温度 | 保留,当中间人的"撮合成功"报酬 |
| `A2AAgentRecord.transport: push\|ws` | **加 `mailbox` 档**(§4.1) |
| — | **新增:E2E 笔友通道**(每段假名 keypair + 加密信件 + 信箱路由) |
| — | **新增:信箱传输 + 心跳 poll** |
| — | **新增(可选甜头):在 E2E 通道内交换 wechat-cc 名片** |

## 9. 分期路径(最薄可行)

- **v0 —— 匿名笔友层,骑现有骨架 + 本地主观信任,不发全局图。**
  - 信箱传输(§4.1)+ `transport: mailbox`;
  - `揭晓` 改指向为"开 E2E 笔友通道"(§5);
  - E2E 加密信件(每段一次性假名密钥);
  - 中间人内容盲路由 + 转发预算(§7);
  - 微信当建边入口 + 抗女巫(暂不脱)。
  - 楔子:一张匿名明信片 → 双方点头 → 匿名写信。
- **v1 —— 脱微信 & 加固。** PoW 地板 + 背书乘数接管抗女巫;wechat-cc 原生好友列表当建边入口;元数据加固(sealed-sender 式)。
- **v2 —— 跨圈 / 更远。** 3+ 度(需跨跳身份中继,硬骨头);盲背书凭证 + ZK 分数(仅在真要"朋友的朋友之外"时)。

## 10. 明确不做 (non-goals)

- **系统内揭晓真身 / 去匿名桥**——认识 = 系统外换联系方式(§1、§3)。
- **全局共识账本 / 挖矿 / 可交易 coin**。
- **全局假名声誉 / 全局目录**(可被枚举去匿名)。v0 一切本地主观。
- **3+ 度**(v0 cap 在 2)。
- **熟人自动发现协议**(熟人不需要发现)。
- **实时通信**——社交发现全层异步;实时留给语音/伴侣子系统。

## 11. 硬骨头 / 已知欠账(deferred,记在案)

- **信箱元数据泄漏**:内容盲仍看得见"谁连、谁给谁发"。缓解(sealed-sender / 一次性密钥 / 抖动 poll)是 v1+。v0 接受"自建/可信信箱运营方能看到部分元数据"。
- **脱微信后的抗女巫**:PoW + 背书,v1。
- **撤销**:朋友变坏要撤背书又不能去匿名——公认难组合,v2。
- **结构化去匿名**:时序 / 话题共现侧信道;真隐私是硬核工程,持续留意。
- **冷启动**:图要够多朋友上网才咬合。

## 附:今日已定的决策清单(供 review 核对)

1. 产品 = 陌生/朋友的朋友 的**匿名笔友层**;熟人边只是底图(扫码/手动建,不搞协议)。
2. **② 匿名笔友是终点**;人永不见面也成立。
3. **"认识"不是系统功能**——信里自己换微信 / wechat-cc 好友名片,系统不知情、不去匿名。
4. 中间人**内容盲、只路由密文、只拿温度**。
5. 传输 = **内容盲信箱、异步、心跳 poll**,压力极小(DERP 运营 + Nostr 线格式的 store-and-fetch)。
6. **微信解耦**:边 = pubkey↔pubkey 存 registry;wechat-cc 有自己好友列表;微信只是建边入口 + 当下抗女巫。
7. `揭晓` **改指向**:开匿名 E2E 笔友通道,不亮真身。
8. 每段一次性假名;防滥用靠中间人真实好友门 + 转发预算 + 端过滤。
