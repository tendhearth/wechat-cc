# v0.5.15 — silence means silence (companion tick deliberation no longer leaks)

**Date**: 2026-05-09
**Tag**: `v0.5.15` + `desktop-v0.5.15`
**Scope**: One-bug fix release. Companion tick was leaking Claude's internal "I've decided not to push" deliberation as visible WeChat messages. Two-line prompt strengthening + a test pin to keep it strengthened.
**Predecessor**: [v0.5.14 — chatroom metadata + daemon config insulation + update flow](2026-05-08-v0.5.14.md)

---

## TL;DR

User reported AI persona ("张凌赫") sending messages like:

> 这次我选择不主动发消息。
> 你刚在排查 504 问题，我先不打断，等你主动反馈进展再接着帮你。

These should never have been sent. The companion tick's contract is "call `reply` to push, or stay silent — silence means nothing happens." Claude was treating "decide not to push" as something that itself needed to be communicated, and was calling `reply` with the explanation.

Root cause: the system prompt's tick guidance and the per-tick wake-up text both said "decide push or stay silent" without explicitly forbidding visible assistant text on the silent branch. Fix tightens both prompts; a regression test pins the wording.

No code path changes — pure prompt strengthening. Aligns with the project's thin-harness principle (prompt edit before structural change).

## What changed

- **`2b89dfc` `fix(companion): silence means silence — strip leaked deliberation from tick`**
  - `src/core/prompt-builder.ts:184` companionSection — adds explicit clause: "不打扰 = 不调 reply 工具，**也不要产生任何可见的 assistant text**（不要解释你为什么不打扰、也不要总结你的推理）。沉默就是沉默。"
  - `src/daemon/wiring/tick-bodies.ts:47-52` pushTick text — restructured to explicit binary: "要 push: 调 reply / 不 push: 直接结束这一轮，**不调用 reply**，**也不产生任何 assistant text**——不要解释你为什么不打扰、不要总结你看到的状态。"
  - `src/core/prompt-builder.test.ts` — new test asserts both `不要产生任何可见的 assistant text` and `沉默就是沉默` are present, so future prompt edits can't quietly weaken the guidance.
  - Defense in depth: system prompt is one-time at session start; tick text is per-wake. Repeating close to the decision point mitigates Claude misreading the system prompt under heavy intervening context.

- **`b1f739c` `docs(spike): companion eval harness — trajectory + replay + LLM-as-judge`**
  - New spike doc at `docs/spike/2026-05-09-companion-eval-harness.md`. Designs a regression-test framework for current companion implementation (04-22 + 04-24 + 04-29 specs) — synthetic multi-session trajectories with probes, time-compressed replay, blind pairwise LLM judging on five dimensions.
  - Two trajectories (`tech_stress_followup_v1`, `emotional_care_v1`) fully fleshed; six more listed for Week-1 build-out.
  - Explicitly out of scope: comparing alternative architectures (SQLite / Honcho / typed schemas — withdrawn in design discussion as transient harness).
  - Implementation pending; doc landing now so the design is referenceable from project memory and future commits.
