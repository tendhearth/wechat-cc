# v0.5.9 — chatroom is now a persistent session, moderator stays neutral

**Date**: 2026-05-06 (later same day as v0.5.8)
**Tag**: `v0.5.9` + `desktop-v0.5.9`
**Scope**: Re-think of /chat semantics after dogfooding v0.5.8. **No breaking changes** — same slash commands, different internal model.
**Predecessor**: [v0.5.8 — codex actually answers + chatroom is moderator-driven](2026-05-06-v0.5.8.md)

---

## Why this exists

v0.5.8 made `/chat` moderator-driven (good) but the moderator was instructed to **manufacture disagreement**:
- Round 2 was a "forced rebuttal turn"
- Speakers were forbidden from saying "基本同意"
- The framing assumed every conversation should turn adversarial

Real conversations aren't always adversarial:
- **Compare X vs Y** — sure, disagreement is valuable
- **Code review** — usually two AIs catch *different* issues (supplementary, not opposed)
- **Philosophical / emotional Q** — user wants two thoughtful presences, not a debate club
- **Quick fact** — user wants the answer, not 4 rounds of forced theatre

User pushback after dogfooding: "双方不一定是对立面，也可以是补充。我感觉不应该设置这些workflow，而是订好边界即可。" v0.5.9 follows that.

Plus a deeper architectural shift: `/chat` was *per-user-message* in v0.5.8 (each inbound triggered a fresh 1-4 round chatroom with empty history). User pointed out the cleaner mental model is **persistent chatroom**: once you enter `/chat`, both AIs are "in the room" until you leave; each new message extends the same conversation.

| | 改动 |
|---|---|
| **Persistent chatroom history** | New `chatroomHistories: Map<chatId, ChatroomEntry[]>` in the coordinator. Each user msg appends to the existing history; moderator sees the whole prior sequence (user msgs + speaker turns interleaved). Cleared when user switches mode away from chatroom (`/cc`, `/codex`, `/solo`). In-memory only — daemon restart resets the moderator's view (speakers' SDK sessions still continue, minor inconsistency, low cost) |
| **Moderator instructions: boundaries, not workflow** | Removed "round 2 = forced rebuttal", "must find weak points", "no '基本同意'". New framing: judge the actual relationship between speakers (agreement / supplementary / disagreement / verification), pick `prompt` accordingly. Explicit ban on **manufacturing virtual opposition**. Filler turns ("basically agree + minor补充" with no new content) → moderator should `end` instead of forcing another round |
| **🎯 final-round emoji marker** | Replaced the heavy `[Codex · 终局]` framing with a single 🎯 emoji prepended to the synthesis turn's body. Reads as "this is the takeaway / verdict" without the video-game-finale tone the user flagged |
| **Plain-text coda still enforced** | `[chatroom 模式]：please reply in plain text, do not call reply tool` still appended to every speaker's prompt. Keeps the visual rhythm consistent — both speakers' turns flow through daemon's `[Display] text` forwarding rather than mixing with MCP-reply-tool turns |

## Resolved Q&A from the design discussion

The design talks before this commit settled:

| Q | Decision |
|---|---|
| 1. User's `@<peer>` in their msg as routing override? | **Skipped** — moderator decides everything, scope tighter |
| 2. History upper bound (truncate / summarize / unbounded)? | **Unbounded for now**. Realistic chat session = O(100s) of entries × O(200 chars) = ~20-40KB to moderator prompt. Haiku handles it. Add cap later if it bites |
| 3. `/stop` semantics under persistent chatroom? | **Just abort current dispatch**. Stay in /chat. Use `/cc` `/codex` `/solo` for explicit mode change |
| 4. Persist chatroom history across daemon restart? | **No**. In-memory only. Speakers' SDK sessions continue independently |
