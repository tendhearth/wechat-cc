# v0.6.1 — Inline quoted-message content, memory-profile redesign, green CI

**Date**: 2026-06-03
**Tag**: `v0.6.1` + `desktop-v0.6.1`
**Scope**: A small patch on top of v0.6.0 — one user-facing companion feature (the agent now sees the full content of a quoted/replied-to WeChat message), a desktop memory-profile overview redesign, and a CI cleanup that takes `master` back to fully green.
**Predecessor**: [v0.6.0 — Multi-provider tiers, dashboard reconnect/diagnose, companion proactivity](2026-05-29-v0.6.0.md)

---

## TL;DR

1. **Quoted-message content reaches the agent.** When a user quotes/replies to an earlier message, the daemon now surfaces the **full** quoted text to the agent as a structured `<quote type="…">…</quote>` element, instead of the old truncated `[引用: …]` snippet and a meaningless `quote_to` id.
2. **Desktop memory-profile overview redesign.** A refreshed memory pane in the desktop app.
3. **CI is green again.** Two pre-existing failures (since v0.6.0) fixed: stale tier e2e tests and a Windows-only `EBUSY` from a leaked db handle.

---

## Companion — inline quoted content

- ilink delivers the quoted message inline in `ref_msg` (it carries **no** stable id, and there is no message store), so the right design is to surface that content, not build a lookup tool. `parseUpdates` now extracts the richest available text (`message_item.text_item.text` → `unsupported_item.text` → `title`) plus a type label, and `formatInbound` renders it as `<quote type="text|image|voice|file|…">原文</quote>` at the **start** of the message body so the agent reads the context before the user's new line.
- The dead `quote_to="<id>"` attribute is gone; the system prompt now documents the `<quote>` element to the agent.
- A degenerate `ref_msg` with neither a known type nor any text is skipped (no empty `<quote>`).
- Scope note: this deliberately ships **no** message store / lookup tool — the original "per-turn msg_id + turn-by-id resolution" backlog framing was infeasible (ilink gives no quoted-message id). General chat scrollback remains a separate, larger, unstarted feature.

## Desktop

- **Memory-profile overview redesign** — refreshed layout/styles for the memory pane (`apps/desktop/src/modules/memory.js`, `styles.css`, new companion asset).

## CI / test reliability

- **Tier e2e tests migrated to the v0.6 strict-mode contract.** The three `user-tier*.e2e.test.ts` acceptance tests still asserted the pre-RFC-05 model where an admin chat gets a *dangerous* SDK spawn in strict mode. Post-RFC-05 (and the 2026-05-26 tier-policy change) admin gets the *safe* spawn shape in strict mode and differs from guest via tier deny/relay — only `--dangerously` yields the dangerous shape. Tests now assert the real contract (claude: both `default`, guest denies `Bash`/`Write`/`Edit`; cursor: both sandboxed; codex: admin `workspace-write`, guest `read-only`). **Product code unchanged.**
- **Windows `EBUSY` fixed.** `cmdAgentActivity` opened a SQLite handle and never closed it; on Windows an open handle blocks file deletion, so the CLI tests' `rmSync` cleanup threw `EBUSY`. Now wrapped in `try/finally → db.close()` (+ a test `seed()` helper). Lesson recorded: any `openWechatDb`/`openDb` caller must close its handle.
- The e2e fake ilink server now serves `getconfig`/`sendtyping`, removing a misleading `[TYPING] … getconfig 404` log on every dispatch (no functional effect — pure noise).

## Upgrade notes

- No config migration. `sessions.json` / `wechat-cc.db` carry over.
- No behavior change for operators — the tier/permission model is exactly as shipped in v0.6.0 (the e2e change is test-only). Carries forward all v0.6.0 known limitations.
