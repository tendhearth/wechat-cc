# v0.5.14 — chatroom metadata + daemon config insulation + update flow that actually works

**Date**: 2026-05-08
**Tag**: `v0.5.14` + `desktop-v0.5.14`
**Scope**: Reliability pass driven by a single user-reported `[Claude] API Error: 404 model: opus` incident in chatroom mode. Excavating the root cause surfaced a chain of framework-level config / metadata-flow gaps that all shared the same shape — implicit data flowing across a boundary without an explicit guarantee. Eight commits closing the chain, plus a full e2e suite covering all 4 chat modes so none of them silently regress again.
**Predecessor**: [v0.5.13 — memory section is now AI's mental model, not a database](2026-05-07-v0.5.13.md)

---

## TL;DR

Three classes of bugs, all with the same root cause shape ("X paraphrases / inherits Y silently"):

1. **chatroom moderator drops user-message metadata** — speaker session never sees `[image:/path]` or `chat_id` because haiku-4-5 paraphrases them away when generating each speaker's prompt. Fixed by re-injecting at the coordinator layer regardless of moderator output.
2. **daemon inherits interactive CLI config silently** — user's `~/.claude/.claude.json` model alias `opus[1m]` worked in their interactive Claude Code session but the daemon-spawned subprocess mis-resolved it and sent literal `"opus"` to the API → 404 on every chatroom dispatch. Fixed by pinning Claude SDK options + trimming `settingSources` + sanitizing systemd env.
3. **`wechat-cc update` didn't actually update the running daemon** — only ran `git pull` + `bun install`, never recompiled the binary that systemd's `ExecStart` points at. Fixed in two passes: rebuild on every update with new commits, then sentinel-based detection so a stale binary still triggers rebuild even when `git` is at HEAD.

Plus the first comprehensive e2e suite for the daemon — 12/12 covering solo / parallel / primary_tool / chatroom × text + image + restart-persistence.

## What changed

### Chatroom metadata flow

- **`f7acca0` `fix(chatroom): inject attachment markers into dispatched speaker prompt`**
  Solo / parallel modes dispatch `format(msg)` directly so `[image:/path]` reaches the speaker. Chatroom funnels through a haiku-4-5 moderator that paraphrases the user msg ("用户发了张图片") and silently strips structural markers — the speaker has Read/Bash but no path. Fix at `src/core/conversation-coordinator.ts:300-313`: unconditionally append `msg.attachments` markers to the dispatched prompt, deduped against whatever the moderator did happen to keep.

- **`b69973f` `fix(chatroom): inject [chat_id:xxx] so speaker can route memory_* tools`**
  Same root-cause shape: `<wechat chat_id="...">` envelope is in moderator history but its generated prompt drops it. Speaker can't namespace memory writes (`memory_write('chat-xxx/profile.md')`) or call `set_user_name(chat_id, ...)`. Fix extends the same coordinator block to also inject `[chat_id:msg.chatId]`.

- **`c096756` Bug B — delegate_<peer> prompt forwarding** (documentation-only)
  When a primary_tool primary calls `delegate_codex({prompt})` it writes the prompt itself; same paraphrasing trap as the moderator. Added explicit instruction in `prompt-builder.ts` `delegateSection`: "must copy `[image:/abs/path]` markers verbatim; if you only describe the attachment, the peer can't open it."

### Daemon config insulation

- **`e6f40f5` `fix(bootstrap): pin Claude model in SDK options, insulate from .claude.json`**
  Original incident trigger. Daemon's `sdkOptionsForProject` had no `model` field, so the spawned Claude Code subprocess inherited `~/.claude/.claude.json`'s `"opus[1m]"` alias. Claude Code 2.1.133 mis-parsed that under SDK mode and sent literal `"opus"` to the API. Codex side already pinned model from `agent-config.json`; this closes the asymmetry. `loadAgentConfig` now preserves the `model` field for both providers; bootstrap defaults to `claude-opus-4-7` when unset.

- **`c096756` `settingSources: ['user','project','local']` → `['project','local']`**
  Same theme: user-global `~/.claude/settings.json` (`effortLevel`, `alwaysThinkingEnabled`, custom mcpServers, model alias preferences) is meant for the human's interactive CLI and shouldn't bleed into a long-running headless daemon. Project + local still load so per-cwd `.claude/` setups continue to work.

- **`c096756` systemd unit env sanitization**
  `~/.config/systemd/user/wechat-cc.service` now emits explicit `Environment="CODEX_MODEL="` and `Environment="WECHAT_AGENT_PROVIDER="` clears. Without these, a `CODEX_MODEL=...` exported in the user's `~/.bashrc` would silently override `agent-config.json`'s pinned model — same class of bleed-in as the `.claude.json` incident, just via a different vector.

### Update flow

- **`3707980` `fix(update): rebuild compiled binary so daemon actually picks up new code`**
  Pre-fix `wechat-cc update` only ran `git pull` + `bun install`. The compiled `wechat-cc-cli` binary that systemd's `ExecStart` points at was never refreshed, so any TypeScript fix landed in master but stayed invisible to binary-mode services. Fix detects binary mode by parsing the installed service file (systemd unit ExecStart on Linux, launchd plist on macOS, scheduled task on Windows). When the first arg basename is `wechat-cc-cli(.exe)?`, update compiles to a sibling `.tmp-pid-ts` and atomically renames over the target. A failed rebuild surfaces `rebuild_failed` and best-effort restarts the daemon on the OLD binary — same "WeChat must not go silently dark" contract as pull/install failures.

- **`0a94357` `fix(update): rebuild stale binary even when git is at HEAD (sentinel-based)`**
  `3707980` only rebuilt on `behind > 0` — fast-path for `git up-to-date` returned `daemon=noop` even if the running binary lagged the local HEAD. Sentinel approach: `<binaryPath>.commit` written atomically alongside the binary on every successful rebuild stores the source SHA. `analyzeUpdate` compares to local HEAD and surfaces `binaryStale=true` when they diverge; `updateAvailable` becomes `behind > 0 || binaryStale`. Pre-fix binaries (no sentinel on disk) are treated as stale on the first update post-upgrade — first run installs the sentinel, subsequent runs short-circuit cleanly.

### Schema discipline (P1.A second-pass)

- **`c16f009` `fix(p1a): close two schema-discipline gaps in CLI subcommands`**
  - `wechat-cc logs --json` called `LogsOutput.parse(result)` before checking `!result.ok`, so a permission error on `channel.log` threw a `ZodError` instead of exiting cleanly.
  - `wechat-cc install-progress --json` skipped the schema entirely, streaming raw file contents to the desktop wizard. A stale/truncated file leaked garbage past the producer-side parse contract that P1.A established.
  Both fixes extract a pure helper (`formatLogsForCli`, `readInstallProgress`) so success/error split is enforced in one place.

- **`5fb46ef` `fix(schema): sync UpdateApplyOutput / UpdateCheckOutput with new fields`**
  Code-review caught the same class of P1.A drift the original P1.A work was meant to prevent: today's `update.ts` changes added `rebuildRan` (required) and `binaryStale` / `binaryPath` (optional) to the runtime types, but the matching Zod schemas weren't updated. Added the fields + a regression test that explicitly rejects `ok:true UpdateApplyOutput` missing `rebuildRan` so a future TS-type-without-Zod-update drift fails CI immediately.

### E2E coverage

- **`f776cba` `test(e2e): cover all 4 chat modes + regression for 2026-05-08 metadata bugs`**
  Expands the daemon e2e suite from 2 (smoke + solo text) to 12, organized in 3 tiers:
  - **Tier 1** (T1.1, T1.2, T1.3): direct regression for the chat_id / image-marker drops + solo image baseline
  - **Tier 2** (T2.1, T2.2, T2.3): every chat mode dispatches correctly — solo codex, parallel /both with [Display] forwarding, primary_tool with secondary as delegate-only
  - **Tier 3** (T3.1, T3.2, T3.3): `/codex` slash-command flips mode for next msg; reply tool produces 1 outbound (not 2 with fallback double-firing); chatroom mode survives daemon stop+start via SQLite

  Infrastructure additions: `fake-media.ts` (vi.mock for `../media` so attachments materialize to local stub files), `installFakeModerator` (chatroom haiku decisions), tool-call → internal-api bridge in fake-sdk so `reply` etc. actually produce outbound msgs, `harness.sendImage()`, `stateDirOverride` for restart tests.

## Migration

- **CLI users**: `wechat-cc update` now does the right thing. First run will detect the missing sentinel + rebuild the binary unconditionally; subsequent runs short-circuit cleanly when binary tracks HEAD.
- **Desktop users**: download the new bundle from the release page, follow the standard `desktop-v0.5.x` migration (stop service / uninstall / reinstall). Re-scanning QR not required — `~/.claude/channels/wechat/` state directory is untouched.
- **Service file regeneration**: the new `Environment="CODEX_MODEL="` / `Environment="WECHAT_AGENT_PROVIDER="` clears only land if you re-run `wechat-cc service install` or its desktop wizard equivalent. Existing systemd units continue to work without the clears (just no protection against shell rc bleed-in until reinstalled).
