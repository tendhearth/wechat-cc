# v0.6.2 — Multi-chat navigation + memory-profile overview

**Date**: 2026-06-04
**Tag**: `v0.6.2` + `desktop-v0.6.2`
**Scope**: Two user-facing desktop features — per-contact navigation in the sessions pane, and a redesigned memory-profile overview — plus a CLI surface for browsing sessions per contact and a band of correctness/hardening fixes.
**Predecessor**: [v0.6.1 — Inline quoted-message content](2026-06-03-v0.6.1.md)

---

## TL;DR

1. **Multi-chat navigation in the sessions pane.** A contact sidebar lets you browse each WeChat contact's sessions separately; the existing transcript view is unchanged. Single-contact setups are unaffected (the sidebar stays hidden).
2. **Memory-profile overview redesign.** The desktop memory pane leads with a profile overview; raw observations/milestones and memory files move into collapsible folds.
3. **New `sessions list-chats` CLI + `--chat` filter** on `list-projects`/`read-jsonl`/`delete` — the backend that powers per-contact browsing (and a fix for a destructive `delete` bug).

---

## Multi-chat navigation (sessions pane)

The sessions pane previously listed one row per project alias and silently collapsed away the `chat_id` dimension, so when more than one WeChat contact used the same project alias, only the most-recent was reachable. v0.6 shipped per-chat session isolation (`(alias, provider, chat_id)`); this release surfaces it in the UI.

- **Contact sidebar** — lists contacts that have sessions (name from `last_user_name`, avatar reused), most-recent first; selecting one filters the session list to that contact. Mirrors the memory pane's sidebar.
- **Single-contact grace** — when there's only one contact (the common case), the sidebar is hidden and the pane looks exactly as before. No regression for single-chat users.
- **Selection survives auto-refresh** — the 30s list refresh and the 4s transcript refresh both keep the selected contact.
- **CLI backend** — new `sessions list-chats --json` (contacts with sessions), and an optional `--chat <chat_id>` filter on `list-projects` / `read-jsonl` / `delete`. The no-`--chat` paths are unchanged, so existing consumers keep working.
- **Scope** — chat × alias only; no multi-account dimension (single bot login assumed). Search stays global across contacts (per-contact search is a deferred follow-up).

### ⚠️ Bug fix — `sessions delete <alias>` was deleting *every* contact's session

Before this release, deleting a project alias from the dashboard removed the session row for **every** chat under that alias, not just the visible one. `delete --chat <chat_id>` now scopes the deletion to the selected contact; the no-`--chat` CLI path retains the old delete-all behavior for non-UI callers.

## Memory-profile overview redesign

The desktop memory pane now leads with an always-visible **profile overview** (a synthesized read of the companion's notes), with the raw **observations/milestones** and the **memory-file browser** tucked into collapsible folds under a "记忆来源" panel. Decisions (events) load on first expand.

Hardening applied during review of the redesign: fixed a render race (the profile rendered before observations finished loading), corrected the artboard scale-to-fit so it no longer overflows at narrow viewports, escaped all dynamic profile output, and made the decisions fold load once instead of on every expand.

## Reliability & test infra

- **Tier e2e tests migrated to the v0.6 strict-mode contract** (they had asserted the pre-RFC-05 "admin == dangerous spawn" model) — product code unchanged.
- **Windows `EBUSY` fixed** — `cmdAgentActivity` now closes its SQLite handle (an open handle blocked file deletion on Windows). Plus a 15s timeout on the first `buildBootstrap` test to absorb cold-import cost on slow Windows CI runners.
- The e2e fake ilink server serves `getconfig`/`sendtyping` (kills a misleading `[TYPING] 404` log); CLI `--help` documents the new `sessions list-chats`/`--chat` surface; the desktop sessions pane gained an inline export-error strip.

## Upgrade notes

- No config migration. `sessions.json` / `wechat-cc.db` carry over.
- Upgraders from v0.5.x whose sessions migrated under `chat_id='_legacy'` will see a single "（早期会话）" contact in the new sidebar.
- No daemon/permission behavior change from v0.6.1. Carries forward all prior known limitations.
